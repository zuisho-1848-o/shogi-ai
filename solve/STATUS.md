## 判明した課題

### 1. 速度がボトルネック
- 1,545,093ノードに127秒 ≈ **12,000 nodes/sec**
- 原因: [`core/board.py:49`](../core/board.py) の `shogi.Board(self._board.sfen())` が毎手でSFEN文字列変換を挟んでいる
- 深さを増やすと指数的に悪化する（depth=7 以上は実用困難）

### 2. `unknown` 局面が残る
- depth=5 では解決できない局面がある（後手LOSS深さ依存シード: 476k nodes でも unknown）
- 探索深さを増やすか、抽象化で合流させるか

### 3. 後退解析の方向性
- 現在は **前向き DFS**（局面から先を探索）
- 本来の後退解析は **逆算**（詰み局面から1手前を生成）
- 逆算の方が「詰みに関係する局面のみ」を効率よく集められるが、将棋の逆算（特に打ち駒）は実装が複雑

### 4. 抽象化はまだ未実装
- 同型局面（座標シフト、左右対称）の同一視が0
- 「詰みに無関係な駒」の省略も未実装

## ネクストアクション（優先順）

### Phase 2: 速度改善（難易度: 中）

**Option A: ビットボード化**
- 盤面を整数のビット列で表現、コピーが O(1)
- 実装コスト大きいが効果絶大（100x以上の高速化見込み）

**Option B: python-shogi の copy() を使う**
- `shogi.Board` に `copy()` があるか確認 → あれば即効
- SFEN経由より高速なはず

**Option C: 探索順の最適化**
- 詰み形に近い手（王手手）を優先して探索
- 枝刈り効果で実質的なノード数を減らす

→ **まず Option B を試す（最小コスト）**

### Phase 3: 基本的な抽象化（難易度: 中）

1. **左右対称の同一視**: 盤面を左右反転して正規化 → 約1/2に削減
2. **座標の相対化**: 玉の位置を原点として他の駒を相対座標で表現
3. **無関係駒の除外**: 詰みに関与しない駒を除いた抽象SFEN → 合流率を計測

実験として「抽象化前後のDB合流率」を計測するスクリプトを作る。

### Phase 4: 体系的な後退解析（難易度: 高）

- **詰み局面の網羅的収集**: 既存の詰め将棋DB（詰将棋めとろ、やさしい詰め将棋etc.）をパース
- **純粋な後退解析**: 詰み局面から1手逆算して「詰ます直前」局面を生成する関数
- **千日手の本格対応**: 繰り返し検出 + ラベルの特別扱い

### Phase 5: パターン抽出・理論化（難易度: 高）

- ラベル付き抽象局面から決定木でルール抽出
- 「○○の形は先手必勝」という命題を自動生成
- 人間が読める将棋定理の出力
