# 将棋を「解く」プロジェクト：後退解析 × 盤面抽象化

## 目的

将棋の盤面に対して、**NNUE や機械学習による評価値近似ではなく**、
「この局面は先手必勝 / 後手必勝 / 不明」という確定ラベルを付与し続けることで、
いつか将棋の初期局面に正解ラベルを与えることを目指す。

---

## コアアイデア

### 1. 詰み局面から後退解析する

- 詰み（終端局面）は勝ち/負けが確定している → ここをスタート地点にする
- そこから「1手前の局面」を逆算し、合法手を全列挙して勝ち/負けのラベルを伝播する
- この後退解析（Retrograde Analysis）を繰り返すことで、確定局面のDBを拡大する

```
詰み局面（負け確定）
  ↑ 1手前
全合法手を列挙 → その全てが「負け確定」局面へ繋がるなら → 1手前も「勝ち確定」
                → 1つでも「勝ち確定」へ繋がる手があるなら → 1手前は「負け確定」（相手最善手を踏む）
```

### 2. 局面の抽象化

**核心:** 盤面には「詰みに無関係な駒」が多数ある。それを除いた抽象表現を使えば、
多くの異なる具体局面が「同じ抽象局面」に合流し、DBサイズが劇的に減少する可能性がある。

抽象化の方向性（試行する順番）：
- **駒省略**: 詰みに直接関与しない駒を削除
- **駒種汎化**: 具体的な駒（金/銀/桂）を「防御駒」「遠隔駒」などにグルーピング
- **座標汎化**: 絶対座標 → 玉からの相対座標に変換
- **対称性**: 盤面の左右対称を同一視
- **持ち駒抽象化**: 持ち駒の多/少を数値ではなくカテゴリで表現

### 3. 合流検出（枝刈り）

- ラベル済みの抽象局面に到達したら、そこで探索を打ち切る
- 有向非循環グラフ（DAG）として局面を管理する（千日手対応要検討）

---

## フロー

```
[詰み局面DB]
     ↓ 後退解析（1手前展開）
[勝ち/負け確定局面DB]
     ↓ 抽象化関数を適用
[抽象局面DB] ← 合流検出で刈り込み
     ↓ さらに後退解析
     …（繰り返し）
     ↓
[初期局面に近づく]
```

---

## 「新理論の発見」という目標について

ラベル付き抽象局面を大量に蓄積した後、以下を試みる：

- **パターン抽出**: 「玉の近くに○○があると先手必勝」などのルールをデータから発掘
- **局面の等価クラス分類**: 異なる具体局面が同一の勝敗を持つ共通構造を発見
- **証明可能な命題の生成**: 「Aの形から最善手を打てば必ず詰む」という命題を自動生成

これは NNUE のような「評価値の近似」ではなく、**確定的・証明可能な知識**の構築を目指す。

## 現在の限界と課題

| 課題 | 内容 |
|------|------|
| 状態空間の巨大さ | 将棋の局面数 ≈ 10^226、完全な後退解析は不可能 |
| 抽象化の妥当性証明 | 「この駒は無関係」と言えるかの証明自体が難しい |
| 千日手・持将棋 | 引き分けになる局面の扱いが複雑 |
| 持ち駒の表現 | 将棋固有の持ち駒システムが抽象化を困難にする |

## 既存アプローチとの違い

| 手法 | 本プロジェクトとの違い |
|------|----------------------|
| NNUE / 機械学習 | 評価値は「近似」。本PJは確定ラベルを求める |
| 詰め将棋ソルバ | 詰み局面限定。本PJはそこから拡張する |
| エンドゲームTB | チェスでは実績あり（Nalimov TB など）。将棋版の発展形 |
| アルファゼロ型 | 自己対局による学習。本PJは証明ベース |

## 実装計画（初期フェーズ）

### Phase 1: 詰み局面の収集と確定DB構築
- 詰め将棋（1手詰〜7手詰）の局面を収集
- 「詰み = 負け確定」「詰み前1手 = 勝ち確定（詰ます手がある）」ラベルを付与
- 確定局面DBを構築（局面ハッシュ → ラベル）

### Phase 2: 後退解析エンジン実装
- 確定局面から1手前の局面を逆算する関数
- 合法手の逆算（何を指せばこの局面になるか）
- 勝ち/負けラベルの伝播ルール実装

### Phase 3: 抽象化関数の実験
- 複数の抽象化関数を実装し、合流率（削減率）を計測
- 最も効果的な抽象化の組み合わせを探索

### Phase 4: パターン抽出・理論化
- ラベル付き抽象局面からルール抽出（決定木・ロジックルール）
- 人間が読める「定理」として出力する試み
